#+TITLE: Audit TODO for test_running.jl

* Overview / audit goal
This checklist is based on the logic in =test_running.jl= and the shared =Processing= code it calls. It is meant for a human researcher reading the script, running it, and deciding which parts of the analysis are reliable, ambiguous, or likely mistaken. It does not assume any specific runtime output in advance.

- [ ] Treat this file as a guide to audit the script's logic, not as proof that any printed result is correct.
- [ ] Keep =test_running.jl= open while using this checklist so you can compare each output block to the code that generated it.
- [ ] Keep notes year by year, because 2014, 2018, and 2022 do not use identical vote or ideology inputs.
- [ ] Separate data-hygiene problems from substantive findings; many apparent inversions could be artifacts of label translation.

* Run the script and capture output
The script prints large tables directly to stdout and mutates global =Processing= paths at startup. Capture a clean run so you can tie any failure to the exact stage where it happens.

- [ ] Run the script from a fresh Julia session and capture both stdout and stderr, for example: =julia test_running.jl 2>&1 | tee out/test_running_stdout.txt=.
- [ ] Note whether the first failure happens in =PART 1. LOAD DATA= or in =PART 2. ANALYZE SEAT DIFFERENTIALS=.
- [ ] If the script stops, record the exact year and subsection that failed before changing any data or code.
- [ ] Save the printed lists of parties, coalition periods, and ideology source names; they are the quickest sanity checks later.

* Validate data loading assumptions
The script constructs several paths by walking upward from =@__DIR__=. If any directory layout changed, the analysis can fail or silently read the wrong files.

- [ ] Confirm the printed =repo_root=, =data_root=, =coalition_json_path=, and =classification_root_dir= point to the intended repository copy.
- [ ] Confirm these election files exist and are the intended versions: =data/raw/electionsBR/2014/party_mun_zone.csv=, =2014/candidate.csv=, =2018/party_mun_zone.csv=, =2018/candidate.csv=, =2022/party_mun_zone.csv=, =2022/candidate.csv=.
- [ ] Confirm the coalition input exists at =scraping/output/partidos_por_periodo.json= and that its period keys look like =YYYY.k=.
- [ ] Confirm ideology source files exist under =scrape_classification/output/classificacao_2023/= and =scrape_classification/output/classificacao_2025/=.
- [ ] Confirm the shared alias table exists at =Processing/data/party_aliases.csv=.
- [ ] Confirm the cabinet-to-election crosswalk exists at =Processing/data/cabinet_to_election_party_crosswalk.csv=, because observed coalition analysis depends on it.
- [ ] Check whether the hard-coded =expected_total_seats = 513= is appropriate for every run you intend to treat as valid.

* Validate party canonicalization and alias mapping
The script canonicalizes party labels in votes, seats, coalitions, and ideology data before joining anything. If those mappings are wrong, every downstream table can look coherent while being substantively off.

- [ ] Review =Processing.normalize_party= and verify that removing accents, punctuation, and spacing does not collapse distinct party labels that should remain separate.
- [ ] Check whether every raw party label seen in vote and seat files maps through =canonical_party_for_election(..., strict = false)= without falling back to accidental equivalence.
- [ ] Audit the local electoral overrides in =test_running.jl=: 2014 =PATRIOTA -> PEN= and the 2022 =AGIR -> AGIR= passthrough.
- [ ] Decide whether any local override is doing substantive work that should instead live in =party_aliases.csv=.
- [ ] Compare the printed party lists from votes and seats for each year before trusting the merged summary.
- [ ] Flag any party label that only appears after normalization and is not obvious from the raw files.

* Validate vote column detection
Vote totals are not taken from one fixed schema. The script selects columns dynamically, and that choice can change the meaning of =valid_total= across years.

- [ ] Confirm 2014 uses =QT_VOTOS_NOMINAIS + QT_VOTOS_LEGENDA= through =ARC.vote_kwargs_for_year(2014)=.
- [ ] Confirm 2018 and 2022 rely on automatic detection and note which columns were actually chosen.
- [ ] If both component columns and a total-vote column exist, manually compare them to see whether the component sum matches the intended valid-vote concept.
- [ ] Check whether any fallback to =QT_VOTOS=, =TOTAL_VOTOS=, or =QT_VOTOS_VALIDOS= would mix valid votes with a broader total.
- [ ] Verify that the script filters to =DS_CARGO == "DEPUTADO FEDERAL"= before aggregating votes nationally.
- [ ] Check whether party totals look plausible after canonicalization, especially for renamed parties.

* Validate seat loading and winner-status logic
Seat counts come from exact winner-status matching in =candidate.csv=. A small mismatch in status labels can propagate into wrong seat totals and false inversion signals.

- [ ] Confirm the raw candidate file uses one of these exact winner labels for elected candidates: =ELEITO=, =ELEITO POR QP=, =ELEITO POR MEDIA=, =ELEITO POR MÉDIA=.
- [ ] Check for winner-status variants that should count as elected but would be missed by exact matching.
- [ ] Verify that party canonicalization in =candidate.csv= uses the same year-specific identity logic as in the vote file.
- [ ] Confirm the national seat total really sums to 513 after the winner filter.
- [ ] Check whether any party has seats but zero votes, or votes but zero seats, and whether that reflects data reality or a join problem.
- [ ] Inspect a few parties manually to confirm the winner count matches the official seat count.

* Validate coalition period linkage to election years
The script links election years to post-election coalition periods by mandate. If that linkage is wrong, the observed coalition analysis is answering the wrong question even if the arithmetic is correct.

- [ ] Confirm the script's linkage is the intended one: 2014 election -> 2015, 2016, 2017, 2018 periods; 2018 election -> 2019, 2020, 2021, 2022 periods; 2022 election -> 2023, 2024, 2025 periods.
- [ ] Decide whether including the mandate-end year in full is theoretically justified for each election.
- [ ] Check whether =coalitions_by_year= is returning periods by direct =YYYY.= prefix instead of by date-window overlap, and decide whether that is acceptable.
- [ ] Compare the printed coalition period keys to =period_start= and =period_end= dates from =coalition_period_windows=.
- [ ] Confirm the raw parties in =partidos_por_periodo.json= are sensible before canonicalization.
- [ ] Flag any period whose dates or label suggest it belongs to a different electoral mandate.

* Validate coalition translation into election-space parties
Observed coalition tables are built in two steps: first canonicalize cabinet-year party labels, then translate them into election-year party identities. This is one of the highest-risk parts of the script.

- [ ] Confirm =Processing.canonicalize_parties(raw_parties; year = coalition_year, strict = true)= resolves every coalition member without ambiguity.
- [ ] Audit =cabinet_to_election_party_crosswalk.csv= for cases where one cabinet party expands into multiple election-year parties.
- [ ] For several periods, manually verify that the final =election_space_parties= all exist in =summary_df.SG_PARTIDO=.
- [ ] Check whether same-label passthrough in =cabinet_parties_in_election_space= is ever incorrectly mixing cabinet-year and election-year party identities.
- [ ] Verify that translated coalitions do not silently gain or lose parties relative to the raw coalition record.
- [ ] Flag any coalition where translation choices materially change the vote total, seat total, or inversion status.

* Validate party summary construction
The central party summary is the base table for both observed coalitions and ideology sweeps. If it is wrong, all later tables inherit the error.

- [ ] Recompute a few party rows manually to confirm =vote_share = votes / total_votes=, =quota = vote_share * total_seats=, and =seat_diff = seats - quota=.
- [ ] Confirm =Processing.party_summary= uses an outer join on =SG_PARTIDO= and replaces missing votes or seats with zero.
- [ ] Decide whether treating missing votes or missing seats as zero is substantively correct for this analysis.
- [ ] Compare the highest positive and negative =seat_diff= parties against raw vote and seat rankings to catch obvious anomalies.
- [ ] Check that the total seats and total votes used later in coalition summaries are exactly those from =summary_df=.

* Validate observed coalition seat differential analysis
The observed coalition table is the script's direct comparison between cabinet coalitions and the election-year party summary. This is where translation mistakes can become substantive-looking findings.

- [ ] For a few coalition periods in each year, manually sum member-party votes and seats from =summary_df= and compare them to =observed_table=.
- [ ] Confirm =vote_majority= means vote share strictly greater than 0.5, and =seat_majority= means seats strictly greater than half of total seats.
- [ ] Decide whether strict > 50% thresholds are the intended theoretical rule or whether ties need explicit treatment.
- [ ] Confirm =candidate_inversion= is defined only as =seat_majority && !vote_majority=.
- [ ] Check whether large coalition =seat_diff= values are robust to alternative party translation choices.
- [ ] Compare adjacent coalition periods to see whether inversion findings are stable or just artifacts of relabeling across periods.

* Validate ideology mapping and ideology ordering
The ideology stage uses source-year classifications and then translates them into election-year party identities. This can drop parties or force historically questionable equivalences.

- [ ] Confirm source selection: 2014 and 2018 use classification 2023, while 2022 uses classification 2025.
- [ ] Review the ideology raw overrides for 2022, especially =CDD -> CIDADANIA= and =PROGRE -> PP=.
- [ ] Review the ideology label overrides for 2014: =AVANTE -> PT DO B=, =DC -> PSDC=, =MDB -> PMDB=, =PATRIOTA -> PEN=, =PODE -> PTN=.
- [ ] Review the 2014 ideology drop list =NOVO=, =PMB=, =REDE= and decide whether those parties are being dropped for good methodological reasons.
- [ ] Check whether any ideology rows are silently skipped because =canonical_party(..., strict = false)= returns =__UNKNOWN__= and no raw override exists.
- [ ] Confirm the script raises an error if translated ideology rows duplicate the same =SG_PARTIDO=.
- [ ] Confirm the script raises an error if any party in =summary_df= has no ideology position after translation.
- [ ] Flag any party that disappears only because of ideology translation, not because it is absent from the election summary.

* Validate ideology sweep logic for minimal majority coalitions
The sweep table is not a search over all coalitions. It only checks contiguous left-to-right blocks in ideology order and stops at the first seat majority from each starting party.

- [ ] Confirm that the sweep operates on ideology-ordered parties joined to =summary_df=, not on observed coalition periods.
- [ ] For several starting parties, manually trace the accumulation to verify the script stops at the first seat majority.
- [ ] Decide whether the note =first seat majority= matches the intended theoretical concept of a minimal majority coalition.
- [ ] Check whether the contiguous-block assumption is substantively defensible for the cases you want to interpret.
- [ ] Inspect all rows with =reached_seat_majority = false= and determine whether the right-tail coalition really never reaches 257 seats.
- [ ] Compare sweep-generated inversions to observed coalition periods before treating them as politically meaningful.

* Validate interpretation of candidate inversions
In this script, =candidate_inversion= is a summary label for an aggregate coalition condition. It should not be read as direct evidence about candidate-level or district-level inversions.

- [ ] Treat =candidate_inversion= as a coalition-level majority test, not as an individual-candidate phenomenon.
- [ ] Keep separate the three concepts the script prints: positive =seat_diff=, seat majority, and seat-majority-without-vote-majority.
- [ ] Check whether any inversion case disappears once alias, crosswalk, or ideology problems are corrected.
- [ ] Note whether inversion appears in observed coalitions, ideology sweeps, or both; those are not interchangeable findings.
- [ ] Ask whether national aggregation is the right level for interpreting proportional representation inversions in the first place.

* Refactor candidates / next code changes
Several risks visible in the script come from local overrides and repeated logic. Even if the current run succeeds, those are prime targets for cleanup.

- [ ] Move durable identity fixes out of =test_running.jl= and into shared alias or crosswalk data where possible.
- [ ] Add a preflight check that all required files exist before any heavy analysis begins.
- [ ] Add diagnostics that print raw -> normalized -> canonical party mappings for votes, seats, coalitions, and ideology inputs.
- [ ] Add a consistency check comparing component votes against total votes when both are available.
- [ ] Make the cabinet-to-election translation step report expansions and passthroughs explicitly.
- [ ] Consider making =expected_total_seats= configurable or validated from metadata instead of hard-coding 513.

* Likely failure modes to investigate first
These are the most plausible substantive problems visible from the script design. Investigate them before spending time on fine-grained interpretation of any inversion result.

- [ ] Alias and label mismatches across election files, coalition JSON, ideology sources, and the cabinet-to-election crosswalk.
- [ ] Parties dropped from ideology ordering because translation fails, because =ideology_drop_labels= removes them, or because the source classification year is a poor historical fit.
- [ ] Manual overrides in =test_running.jl= masking upstream data-hygiene problems that should be fixed in shared normalization code.
- [ ] Coalition-year versus election-year translation mistakes, especially where a cabinet party expands into multiple election parties.
- [ ] Vote-column detection selecting a total-vote field that is not actually the same concept as valid legislative votes.
- [ ] Winner-status exact matching missing legitimate elected records and distorting party seat totals.
- [ ] The hard-coded =expected_total_seats = 513= hiding edge cases when filtering or winner logic is incomplete.
- [ ] The ideology sweep detecting artifacts of ordering and the =first seat majority= stopping rule rather than politically meaningful coalitions.
